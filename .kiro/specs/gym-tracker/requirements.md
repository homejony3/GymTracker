# Requirements Document

## Introduction

A mobile-first web application for tracking gym workouts with a focus on progressive overload. The application supports multiple users, organizes exercises by workout splits (UPPER, LOWER, ARMS), and tracks weight (in kilograms), repetitions, and sets per exercise per session. The app provides persistent storage for historical comparison and suggests weight increases to facilitate progressive overload. It is hosted on a self-managed Linux server (Proxmox VM) and serves EU-based users with appropriate localization.

## Glossary

- **Gym_Tracker**: The web application system as a whole
- **User**: An authenticated person using the application (e.g., the owner and their girlfriend)
- **Workout_Split**: A category of exercises grouped by muscle focus — one of UPPER, LOWER, or ARMS
- **Exercise**: A named physical movement performed during a workout session (e.g., Bench Press, Squat, Bicep Curl)
- **Session**: A single workout event on a specific date, associated with one Workout_Split
- **Set**: A single continuous series of repetitions of an Exercise within a Session
- **Rep**: One complete movement cycle of an Exercise
- **Progressive_Overload**: The gradual increase of weight, reps, or sets over time to stimulate muscle adaptation
- **Weight_Suggestion**: A recommendation generated by the system for the next weight to use on an Exercise based on historical performance
- **Auth_Module**: The component responsible for user authentication and session management
- **Session_Log**: The stored record of all Sets performed for all Exercises within a Session
- **History_Store**: The persistent data layer that retains all past Session_Logs for comparison

## Requirements

### Requirement 1: User Authentication

**User Story:** As a User, I want to log in with my own account, so that my workout data is private and separate from other users.

#### Acceptance Criteria

1. WHEN a User provides a valid username and password combination, THE Auth_Module SHALL authenticate the User and create an active session with a maximum duration of 30 days.
2. WHEN a User provides an invalid username or password, THE Auth_Module SHALL reject the login attempt and display an error message indicating invalid credentials without revealing which field was incorrect.
3. IF a User fails to authenticate 5 consecutive times within a 15-minute window, THEN THE Auth_Module SHALL temporarily lock login attempts for that account for 15 minutes and display a message indicating the lockout duration.
4. THE Auth_Module SHALL maintain separate data isolation between authenticated Users.
5. WHILE a User is not authenticated, THE Gym_Tracker SHALL restrict access to all workout data and functionality and redirect the User to the login screen.
6. IF a User session expires after 30 days of inactivity or the session token becomes invalid, THEN THE Auth_Module SHALL redirect the User to the login screen and discard any unsaved client-side state.

### Requirement 2: Workout Split Management

**User Story:** As a User, I want to organize my exercises into workout splits (UPPER, LOWER, ARMS), so that I can structure my training program.

#### Acceptance Criteria

1. THE Gym_Tracker SHALL provide three predefined Workout_Splits: UPPER, LOWER, and ARMS.
2. WHEN a User selects a Workout_Split, THE Gym_Tracker SHALL display all Exercises associated with that split for the authenticated User, ordered by the date each Exercise was added to the split (most recently added first), up to a maximum of 50 Exercises per split.
3. WHEN a User adds an Exercise to a Workout_Split, THE Gym_Tracker SHALL persist the association between the Exercise and the selected split, allowing the same Exercise to belong to multiple Workout_Splits simultaneously.
4. IF a User attempts to add an Exercise that is already associated with the selected Workout_Split, THEN THE Gym_Tracker SHALL reject the addition and display an error message indicating the Exercise already exists in that split.
5. WHEN a User removes an Exercise from a Workout_Split, THE Gym_Tracker SHALL delete the association and retain historical Session_Logs that reference the Exercise.
6. IF a User selects a Workout_Split that contains no Exercises, THEN THE Gym_Tracker SHALL display an empty state message indicating no Exercises have been added to that split.

### Requirement 3: Exercise Management

**User Story:** As a User, I want to add, edit, and remove exercises within my workout splits, so that I can customize my training program.

#### Acceptance Criteria

1. WHEN a User creates a new Exercise, THE Gym_Tracker SHALL store the Exercise with a name between 1 and 50 characters (trimmed of leading and trailing whitespace) and associate it with the specified Workout_Split.
2. WHEN a User edits an Exercise name, THE Gym_Tracker SHALL update the name across all future Sessions while preserving historical Session_Logs with the original name.
3. WHEN a User removes an Exercise, THE Gym_Tracker SHALL remove the Exercise from the active Workout_Split and retain all historical Session_Logs referencing the Exercise.
4. THE Gym_Tracker SHALL prevent duplicate Exercise names within the same Workout_Split for the same User using case-insensitive comparison.
5. IF a User attempts to create or rename an Exercise with a name that already exists in the same Workout_Split (case-insensitive), THEN THE Gym_Tracker SHALL reject the operation and display an error message indicating the name is already in use.
6. IF a User attempts to create or rename an Exercise with an empty name or a name exceeding 50 characters, THEN THE Gym_Tracker SHALL reject the operation and display an error message indicating the name length constraint.

### Requirement 4: Session Logging

**User Story:** As a User, I want to log my workout sessions with weight, reps, and sets for each exercise, so that I have a complete record of my training.

#### Acceptance Criteria

1. WHEN a User starts a new Session, THE Gym_Tracker SHALL associate the Session with the current date and the selected Workout_Split.
2. WHEN a User logs a Set for an Exercise, THE Gym_Tracker SHALL record the weight in kilograms (accepting values from 0.0 to 500.0 in 0.5 kg increments), the number of Reps (accepting integer values from 1 to 999), and the Set number.
3. THE Gym_Tracker SHALL allow a User to log up to 50 Sets per Exercise within a single Session.
4. WHEN a User completes a Session, THE Gym_Tracker SHALL persist the Session_Log to the History_Store.
5. WHILE a Session is in progress, THE Gym_Tracker SHALL allow the User to edit or delete any Set within that Session.
6. THE Gym_Tracker SHALL display weight values in kilograms with a precision of 0.5 kg increments.
7. IF a User attempts to log a Set with a weight or Reps value outside the accepted range, THEN THE Gym_Tracker SHALL reject the entry and display an error message indicating the valid range.
8. IF a User attempts to complete a Session with no Sets logged, THEN THE Gym_Tracker SHALL prevent completion and display a message indicating that at least one Set must be logged.

### Requirement 5: Session History and Comparison

**User Story:** As a User, I want to view my past workout sessions, so that I can track my progress over time.

#### Acceptance Criteria

1. WHEN a User requests session history, THE Gym_Tracker SHALL display a chronological list of past Sessions ordered from most recent to oldest, showing date and Workout_Split, with a maximum of 50 Sessions per page.
2. WHEN a User selects a past Session, THE Gym_Tracker SHALL display the complete Session_Log including all Exercises, Sets, weights, and Reps.
3. WHEN a User views an Exercise within a Session, THE Gym_Tracker SHALL display the most recent prior Session_Log for the same Exercise performed by the same User for comparison.
4. IF no previous Session_Log exists for the viewed Exercise, THEN THE Gym_Tracker SHALL indicate that no prior data is available for comparison.
5. THE History_Store SHALL retain all Session_Logs indefinitely until explicitly deleted by the User.
6. THE Gym_Tracker SHALL display all dates in EU format (DD.MM.YYYY).
7. IF a User requests session history and no past Sessions exist, THEN THE Gym_Tracker SHALL display a message indicating that no sessions have been recorded.

### Requirement 6: Progressive Overload Suggestions

**User Story:** As a User, I want the app to suggest weight increases based on my past performance, so that I can progressively overload my muscles.

#### Acceptance Criteria

1. WHEN a User starts logging an Exercise in a new Session, THE Gym_Tracker SHALL display the Weight_Suggestion for that Exercise within 2 seconds of the Exercise being selected.
2. THE Gym_Tracker SHALL calculate the Weight_Suggestion based on the most recent Session_Log for the same Exercise where the User completed at least the same number of Sets and Reps per Set as recorded in the Session_Log preceding it.
3. WHEN the most recent Session_Log shows the User completed all Sets with the same or greater number of Reps per Set compared to the Session_Log before it, THE Gym_Tracker SHALL suggest increasing the weight by the configured increment for that Exercise (default: 1.0 kg, configurable per Exercise in 0.5 kg steps between 0.5 kg and 5.0 kg).
4. WHEN the most recent Session_Log shows the User completed fewer Sets or fewer Reps in any Set compared to the Session_Log before it, THE Gym_Tracker SHALL suggest maintaining the same weight as the most recent Session_Log.
5. THE Gym_Tracker SHALL allow the User to override any Weight_Suggestion and log a different weight.
6. IF no prior Session_Log exists for an Exercise, THEN THE Gym_Tracker SHALL display no Weight_Suggestion and allow the User to enter a starting weight freely.

### Requirement 7: Mobile-First Responsive Design

**User Story:** As a User, I want to use the app comfortably on my phone during workouts, so that I can log exercises in real time at the gym.

#### Acceptance Criteria

1. THE Gym_Tracker SHALL render all interactive elements as visible, operable, and without content clipping or overlap on viewports with a minimum width of 320px.
2. THE Gym_Tracker SHALL size all interactive elements to a minimum touch target of 44x44 pixels.
3. THE Gym_Tracker SHALL achieve a Largest Contentful Paint (LCP) of 3 seconds or less on the landing page when tested on a simulated 4G connection (9 Mbps download, 1.5 Mbps upload, 170ms RTT).
4. THE Gym_Tracker SHALL pass all acceptance criteria without errors on the latest two major versions of Safari (iOS) and Chrome (Android).
5. WHILE a User is logging a Session, THE Gym_Tracker SHALL display all form inputs, buttons, and data displays within the viewport width (320px to 767px) without requiring horizontal scrolling.
6. WHEN the device orientation changes between portrait and landscape, THE Gym_Tracker SHALL reflow the layout to fit the new viewport dimensions within 1 second without loss of user-entered data.

### Requirement 8: EU Localization

**User Story:** As a User based in the EU, I want the app to use familiar formats and units, so that I can read data without mental conversion.

#### Acceptance Criteria

1. THE Gym_Tracker SHALL display all weight values in kilograms (kg) with one decimal place using a comma as the decimal separator (e.g., "72,5 kg").
2. THE Gym_Tracker SHALL display all dates in DD.MM.YYYY format.
3. WHEN a User enters a weight value, THE Gym_Tracker SHALL accept both comma and dot as decimal separator characters and interpret them equivalently.
4. THE Gym_Tracker SHALL display time values in 24-hour format (HH:MM).
5. IF a User enters a weight value containing more than one decimal separator or non-numeric characters other than a single separator, THEN THE Gym_Tracker SHALL reject the input and display an error message indicating the accepted format.

### Requirement 9: Multi-User Data Isolation

**User Story:** As a User, I want my workout data to be completely separate from other users, so that I only see my own exercises and progress.

#### Acceptance Criteria

1. THE Gym_Tracker SHALL associate all Exercises, Sessions, and Session_Logs with the authenticated User who created them.
2. WHEN a User queries Exercises, Sessions, Session_Logs, or Weight_Suggestions, THE Gym_Tracker SHALL return only records belonging to that authenticated User.
3. THE Gym_Tracker SHALL prevent any User from accessing, modifying, or deleting another User's Exercises, Sessions, or Session_Logs through any interface, including direct resource identifier manipulation.
4. IF a request references a resource belonging to another User, THEN THE Gym_Tracker SHALL reject the request with a response indistinguishable from a request referencing a non-existent resource.
5. THE Gym_Tracker SHALL enforce data isolation at the query level such that no API response or page view contains references to another User's data.

### Requirement 10: Persistent Storage

**User Story:** As a User, I want my data to be stored reliably, so that I never lose my workout history.

#### Acceptance Criteria

1. THE History_Store SHALL persist all User data (Exercises, Sessions, Session_Logs, and Workout_Split associations) across server restarts and application updates.
2. THE History_Store SHALL use an ACID-compliant relational database system.
3. IF a write operation to the History_Store fails, THEN THE Gym_Tracker SHALL notify the User of the failure, retain the data in the client, and retry the write up to 3 times with a 2-second interval between attempts.
4. THE Gym_Tracker SHALL perform data writes synchronously before confirming success to the User.
5. IF all retry attempts for a write operation are exhausted, THEN THE Gym_Tracker SHALL display a persistent error message indicating the data was not saved and preserve the unsaved data in the client until the User manually triggers a retry or the next successful write.
6. THE History_Store SHALL support automated daily backups that retain at least 7 days of backup history.

### Requirement 11: Self-Hosted Deployment

**User Story:** As the application owner, I want to host the app on my existing Linux server, so that I maintain full control over the infrastructure.

#### Acceptance Criteria

1. THE Gym_Tracker SHALL be deployable on a Linux-based server (Debian 12+ or Ubuntu 22.04+) running on a Proxmox VM using a documented set of steps that results in the application responding to HTTP health checks within 60 seconds of initiating deployment.
2. THE Gym_Tracker SHALL be accessible via HTTPS through a configured domain or subdomain.
3. THE Gym_Tracker SHALL operate as a containerized application or as a standalone service manageable via systemd.
4. WHILE serving up to two concurrent Users performing browsing and logging actions, THE Gym_Tracker SHALL consume no more than 512 MB of RAM.
5. IF the Gym_Tracker process crashes or the host VM restarts, THEN THE Gym_Tracker SHALL restart automatically and become operational within 60 seconds without loss of previously persisted data.
6. THE Gym_Tracker SHALL persist all User data across application restarts and host reboots.
